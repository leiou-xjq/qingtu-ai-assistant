# 青途智伴AI生活助手 - 面试问题手册

## 一、项目与架构问题

### Q1: 请介绍一下这个项目的整体架构

**参考回答**:
```
青途智伴是一个面向大学生的全场景AI生活助手，采用典型三层架构：

1. 前端层：UniApp + Vue3，支持微信小程序、H5、APP多端
2. 网关层：Spring Boot处理JWT认证、统一响应、异常处理、限流熔断
3. 业务层：
   - Controller接收请求
   - OrchestratorAgent编排多Agent
   - SpecialistAgent执行具体任务
4. 数据层：
   - MyBatis-Plus操作MySQL
   - Elasticsearch做向量检索和混合检索
   - Redis + Caffeine做二级缓存

核心模块：
- AI模块：OrchestratorAgent编排 + SpecialistAgent执行 + IntentAnalyzer意图识别
- RAG模块：Elasticsearch做向量存储和混合检索
- MCP模块：统一调度工具、权限控制、审计日志
- 定时任务：Quartz管理早安推送、课程提醒等任务
```

### Q2: 为什么选择Spring Boot作为后端框架？

**参考回答**:
```
选择Spring Boot的原因：

1. 生态成熟：大量第三方库支持，文档完善
2. 自动配置：减少样板代码，提升开发效率
3. 内嵌服务器：打包成jar直接运行，部署简单
4. 社区活跃：问题容易找到解决方案

对比其他方案：
- 裸机Java：需要自己实现IoC、AOP、事务管理等，工作量大
- JPA：复杂查询性能差，SQL优化困难
- GraalVM：性能好但生态不如Spring，排查问题困难
```

## 二、RAG知识库问题

### Q6: RAG的实现原理是什么？

**参考回答**:
```
RAG = Retrieval Augmented Generation（检索增强生成）

原理：
1. 索引阶段：
   - 将知识库文档分块（chunk）
   - 每个chunk调用embedding模型生成768维向量
   - 向量存入Elasticsearch

2. 检索阶段：
   - 用户问题同样生成查询向量
   - ES做KNN向量检索（余弦相似度）
   - 可加关键词过滤（学校分类等）
   - 返回Top-K相关文档

3. 生成阶段：
   - 将检索到的文档作为上下文
   - 构建Prompt: [系统提示] + [知识上下文] + [用户问题]
   - 调用LLM生成最终回答

为什么用RAG：
- 减少幻觉：回答基于真实知识，不是胡编
- 知识更新：只需更新知识库，无需重新训练
- 可溯源：回答可以标注来源
```

### Q7: 为什么选择Elasticsearch做向量检索？不用Milvus或Pinecone？

**参考回答**:
```
选择ES的原因：

1. 功能全面：
   - 支持关键词检索 + 向量检索的混合查询
   - 内置分词器（ik_smart）优化中文
   - 一个引擎解决两种需求

2. 运维简单：
   - 已有MySQL经验，ES上手成本低
   - 可与现有数据源共部署
   - Kibana可视化界面

3. 成本可控：
   - 自托管，无云服务费用
   - 硬件要求不高（单节点即可）

为什么不选其他：

| 方案 | 缺点 |
|------|------|
| Milvus | 需单独部署，运维成本高 |
| Pinecone | 云服务，有成本和数据出境风险 |
| Qdrant | Rust开发，排查问题困难 |

如果未来向量数据超过1000万条，可以考虑迁移到专业向量库。
```

### Q8: 向量检索的具体实现细节？

**参考回答**:
```
实现步骤：

1. Embedding生成
```java
float[] embedText(String text) {
    // 调用DashScope text-embedding-v1 API
    String url = "https://dashscope.aliyuncs.com/api/v1/services/embeddings/text-embedding/text-embedding";
    // POST: {model: "text-embedding-v1", input: {texts: [text]}}
    // 返回: {output: {embeddings: [{embedding: [0.123, ...]}]}}
}
```

2. 存储时写入向量
```java
doc.put("embedding", embedding);  // 768维float数组
esClient.index(IndexRequest.of(i -> i.index("rag_knowledge").document(doc)));
```

3. KNN检索
```java
SearchRequest request = SearchRequest.of(s -> s
    .index("rag_knowledge")
    .size(5)
    .query(q -> q.bool(b -> b
        .filter(f -> f.term(t -> t.field("schoolWebsite").value(school)))
        .should(sq -> sq.knn(k -> k
            .field("embedding")
            .queryVector(queryVector)
            .k(5)
        ))
    ))
);
```

4. 余弦相似度
ES的dense_vector字段配置similarity: "cosine"，自动计算相关性得分。
```

### Q9: 如何优化RAG的检索效果？

**参考回答**:
```
优化方向：

1. 检索效果优化
   - chunk大小：控制在300-500字，过长干扰相关度
   - 混合检索：向量检索 + 关键词检索组合
   - 重排序：对KNN结果用BM25二次排序

2. Prompt工程
   - 系统提示词明确要求基于上下文回答
   - 上下文过长时截断或分段检索
   - 引导模型标注来源

3. 知识库质量
   - 爬取时去除广告、导航等噪声内容
   - 定期清理过时知识
   - 按学校/分类隔离，减少干扰

4. 评估迭代
   - 记录用户反馈（点赞/点踩）
   - 分析bad case针对性优化
   - A/B测试不同检索策略
```

### Q13: 如何保证定时任务不重复执行？

**参考回答**:
```
分布式锁方案：

```java
public class RedisLockUtil {
    public boolean tryLock(String lockKey, int expireSeconds) {
        return Boolean.TRUE.equals(
            redisTemplate.opsForValue()
                .setIfAbsent(lockKey, "1", expireSeconds, TimeUnit.SECONDS)
        );
    }

    public void unlock(String lockKey) {
        redisTemplate.delete(lockKey);
    }
}

// Job中使用
@Override
public void execute(JobExecutionContext context) {
    String lockKey = "task:lock:morning_push:" + userId;
    if (RedisLockUtil.tryLock(lockKey, 300)) {  // 5分钟锁
        try {
            sendMorningPush(userId);
        } finally {
            RedisLockUtil.unlock(lockKey);
        }
    }
}
```

为什么不用数据库：
- 数据库锁查询慢
- 长事务影响其他操作
- Redis SETNX是原子操作，性能高
```

## 五、缓存架构问题

### Q14: 为什么要用二级缓存？

**参考回答**:
```
缓存分层：

┌─────────────────────┐
│  Caffeine (本地)     │  L1
│  进程内，无网络开销   │
└─────────────────────┘
        │ 未命中
        ▼
┌─────────────────────┐
│  Redis (分布式)      │  L2
│  多实例共享          │
└─────────────────────┘
        │ 未命中
        ▼
┌─────────────────────┐
│     MySQL           │
│     持久化          │
└─────────────────────┘

选择Caffeine原因：
1. Spring Cache抽象支持
2. 性能优秀（基于ConcurrentHashMap）
3. 可配置淘汰策略（LRU/WINDOW_LRU等）
4. 统计信息完善
```

### Q15: 缓存一致性问题如何解决？

**参考回答**:
```
缓存更新策略：Cache-Aside（旁路缓存）

1. 读：缓存未命中 → 查DB → 写入缓存
2. 写：更新DB → 删除缓存（不是更新）

为什么删除而不是更新：
- 更新缓存有可能导致数据不一致
- 删除是幂等操作
- 简单高效

课程缓存场景：
```java
// 添加课程后
public void addCourse(CourseDTO dto) {
    courseMapper.insert(dto);
    // 删除缓存，不是更新
    redisTemplate.delete("course:schedule:" + userId);
}

// 查询时
public List<CourseDTO> getSchedule(Long userId) {
    String key = "course:schedule:" + userId;
    List<CourseDTO> cached = redisTemplate.opsForValue().get(key);
    if (cached != null) return cached;

    List<CourseDTO> courses = courseMapper.selectByUserId(userId);
    redisTemplate.opsForValue().set(key, courses, 24, TimeUnit.HOURS);
    return courses;
}
```
```

## 七、MCP多组件协同问题

### Q19: MCP是什么？你们是如何实现的？

**参考回答**:
```
MCP = Model Context Protocol（模型上下文协议）

核心思想：
- 统一管理AI可调用的工具
- 标准化工具调用接口
- 支持权限控制和审计

我们的实现：

1. ToolRegistry - 工具注册中心
```java
Map<String, ToolHandlerData> toolHandlers = new HashMap<>();
toolHandlers.put("rag_search", new ToolHandlerData(
    "rag_search",
    params -> ragTools.search(params),
    ToolPermission.USER
));
```

2. Permission Layer - 权限控制
   - PUBLIC: 公开工具
   - USER: 登录用户
   - ADMIN: 管理员

3. Audit Log - 审计日志
   - 记录工具调用、参数、返回值、耗时

4. 统一执行入口
```java
public CommonResult<?> executeTool(Long userId, String toolName, Map<String, Object> params) {
    ToolHandlerData tool = toolHandlers.get(toolName);
    if (!hasPermission(userId, tool.permission)) {
        return CommonResult.fail("权限不足");
    }
    Object result = tool.execute(params);
    auditLogService.log(userId, toolName, params, result);
    return CommonResult.success(result);
}
```
```

### Q20: Skill和MCP有什么区别？

区别：

## 八、数据库与系统设计

### Q21: 数据库表设计的考量？

**参考回答**:
```
设计原则：

1. 三大范式为基础，适度反范式
   - 基础信息放user表
   - 扩展信息独立表（user_health）
   - 避免大量NULL字段

2. 字段类型选择
   - VARCHAR vs TEXT: 有长度限制用VARCHAR
   - DATETIME vs TIMESTAMP: 需要大范围用DATETIME
   - INT vs BIGINT: 用户量级用INT足够

3. 索引设计
```sql
-- 查询最频繁的字段加索引
CREATE INDEX idx_user_weekday ON course_schedule(user_id, weekday);
CREATE INDEX idx_user_status ON sys_notification(user_id, status);
```

4. 逻辑删除
   - 财务、课程等数据不物理删除
   - deleted字段标识
   - MyBatis-Plus @TableLogic自动处理

5. 避免过度设计
   - 不提前优化
   - 按需加字段
```

### Q22: 如何处理高并发？

**参考回答**:
```
高并发策略：

1. 缓存
   - 热点数据缓存（课程表、用户信息）
   - 多级缓存减少DB压力

2. 异步
   - 爬取、生成笔记等耗时操作异步化
   - 用户无感知等待

3. 限流
   - 接口限流防止刷接口
   - AI调用限流控制成本

4. 数据库优化
   - 读写分离
   - 分表分库（当前规模不需要）
   - SQL优化（避免全表扫描）

5. 分布式
   - 多实例部署
   - 分布式Session
   - 分布式锁
```

class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public LRUCache(int capacity) {
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    public V get(K key) {
        return super.get(key);
    }

    public void put(K key, V value) {
        super.put(key, value);
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry eldest) {
        return size() > capacity;
    }
}

```java
public class RedisLock {
    private RedisTemplate<String, String> redisTemplate;

    public boolean tryLock(String lockKey, String lockValue, long expireSeconds) {
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, expireSeconds, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(result);
    }

    public void unlock(String lockKey, String lockValue) {
        String currentValue = redisTemplate.opsForValue().get(lockKey);
        if (lockValue.equals(currentValue)) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

### 6. 二分查找

```java
public int binarySearch(int[] nums, int target) {
    int left = 0, right = nums.length - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2;

        if (nums[mid] == target) {
            return mid;
        } else if (nums[mid] < target) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }

    return -1;
}
